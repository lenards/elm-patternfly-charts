PatternFly v6 Charts in Elm

Two experiments comparing Elm chart libraries for infrastructure dashboards · OpenStack · Exosphere

terezka/elm-charts

High-level declarative API

Area Chart Bar Chart Stacked Area

React-like element + attributes API. Handles axes, grid lines, legends, and tooltips automatically. Least boilerplate for Cartesian charts (bar, line, area). Think of it as the Victory.js of Elm.

Constraint: No pie or donut chart support — it's Cartesian-only.

gampleman/elm-visualization

Low-level D3-style primitives

Area Chart Bar Chart Donut Chart

Scale, Shape, Axis primitives — the D3.js of Elm. You compose every SVG element manually. More boilerplate but complete control; perfect for non-standard chart types or custom interactions.

Strength: Donut, pie, histogram, force-directed graphs, anything.

Verdict & Recommendation

For Exosphere / OpenStack dashboards: a hybrid approach makes sense. Use elm-charts for area and line charts (CPU, network, memory over time) where its automatic axis handling saves significant boilerplate. Use elm-visualization for donut and pie charts (storage allocation, quota usage) and anything non-Cartesian.

A thin PF.Charts abstraction layer wrapping both libraries with shared color tokens, font settings, and axis styles would give you a consistent PatternFly look without duplicating the styling code in every chart.

Both libraries can match PF v6 design guidelines. The main differentiator is how much you want to control vs. how fast you want to ship.
